# セキュリティと権限

> 目次に戻る → [[index]] ／ 関連 → [[安全第一の哲学]] ・ [[AIの役割と境界]] ・ [[用語集]]

## secret の扱い

- secret は `.env` のみ。`.env.example` に実値を入れない。
- コード・docs・コミットに実クレデンシャル（DB接続文字列/APIキー/トークン/秘密鍵）を書かない。
- 注意文としての「secret」という**単語**は OK、**実値**は NG。この区別を検査でも守る。

## RBAC（役割ベースアクセス制御）

- 定義: `packages/shared/rbac.ts`（10ロール×8アクション、ワイルドカード対応）。
- 使い方: `requireUser()` ＋ `hasPermission(user, resource, action)`（`apps/web/lib/auth/current-user.ts`）。
- 監査系ページは `hasPermission(user, 'audit', 'read')` でガード（新permissionを足さず既存を流用するのが安全）。
- **RBAC定義を安易に変更しない。** 特に AI の権限（`AI_AGENT`/`AI_ASSISTANT`）を広げない。

## 機密ラベルと参照記録

- 機密ラベル: `packages/shared/labels.ts`。
- 監査ログ: `writeAudit`。機密参照（AI含む）: `writeDataAccess`（`apps/web/lib/db.ts`）。

## 外部通信（SSRF対策）

- 外部URLフェッチは `safeFetch` 経由のみ（SSRFガード / allowlist / timeout / サイズ上限）。
- 生の `fetch` で外部に出ない。

## 承認が要る危険操作

- 外部送信 / 契約 / 請求 / 支払 / 削除 / エクスポート / 人事 は `requiresApproval` で承認必須。
- **AIは外部送信・承認・削除を持たない。** 外部送信は `EXTERNAL_SEND_ENABLED=true` ＋ 人間承認時のみ。

## メール営業のコンプライアンス

- 同意管理（`ConsentRecord`）・配信停止（`SuppressionList`）・送信者情報・送信前承認・送信ログ（`OutreachSendLog`）を必須にする。
- AI生成営業文は必ず `OutreachDraft`（下書き）。送信は `/approvals` 承認後、`decideApprovalAction` 経由。
- 送信前に `isSuppressed` で抑止確認。返信に `detectUnsubscribeRequest` を検知したら `SuppressionList` へ追加。
- 大量迷惑送信を助長する実装にしない。

## Google Maps / Places

- 規約違反のスクレイピング禁止。公式API/許諾データ/アップロード/デモのみ。
- Google由来データには `source/placeId/fetchedAt/expiresAt/attributionRequired/cachePolicy` を保持し、帰属表示付きで Google Maps 上に表示。非Google地図に混在させない。

## テナント分離

- 全モデルに `tenantId`。クエリは必ず `tenantId` でスコープ。横断表示は原則しない（必要でも別設計）。
