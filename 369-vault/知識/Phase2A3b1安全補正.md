## 結論（1行）

**main 反映の直前に境界を2つ締め直した — AIは会社方針を一切変更できない・高機密ラベルは参照ログができるまで扱わない。**

## 何を直したか（非エンジニア向け）

1. **AI mutation禁止**: AI社員/AIアシスタントのロールを持つセッションは、権限設定に関係なく会社方針の作成・編集・アーカイブがすべて拒否されるようにした。
2. **ラベルは「通常」「社内限」だけ**: 高機密のラベル（機密・厳秘・役員限など）は、「誰がいつ見たか」を記録する仕組み（writeDataAccess）ができるまで、そもそも作れない・付け替えられないようにした。高機密の方針は編集画面自体が開かない。

## なぜ push 前に直したか（学び）

- **「AIは下書きを作れる」と「AIが会社の方針を書ける」は別物**: 既存 RBAC では AI_AGENT が knowledge:create（下書き原則）を持つ。営業メール下書き等では正しい設計だが、**会社方針は会社の判断基準そのもの**であり、AIが自分の行動規範を自分で書き換えられる構造は危険。rbac を変えず、画面の Server Action 側で「この対象だけ人間専用」と締めるのが最小・安全だった。
- **「作れる機密レベル」は「守れる機密レベル」を超えてはいけない**: 参照ログが無いのに厳秘データを作れるのは順序が逆。機能の解禁順序は「守る仕組み→扱えるデータ」の順。
- **監査→最小補正→検証→記録の型**: 実装が green でも、main 反映前の安全監査で境界を再点検する価値がある。今回の2点はどちらも「動作は正常・境界が甘い」類で、テストでは検出できない。

## 変えなかったもの

- rbac.ts / labels.ts（権限とラベルの定義は無変更＝AI権限の拡大も縮小もしていない）
- externalAiAllowed の封印（create false 固定・update 不変更・UI で true 不可）は doc39 のまま維持
- smoke 13本体制（13/13 green 維持・既存12本無変更）
