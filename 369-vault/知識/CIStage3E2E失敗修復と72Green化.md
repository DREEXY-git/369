## これは何か

- CI（GitHub の自動テスト）に組み込んだ「本物のブラウザで画面を通しで確認するテスト（E2E・全72本）」が、初回実行で **72本中15本失敗**した状態から、**72本全部合格（72 passed / 0 failed）** に到達するまでの修復の全記録（doc131〜143）を1本にまとめたものです。
- 大事な結論: 15本の失敗は**すべて「テストの書き方」の問題**で、**本当のアプリ不具合（TRUE_APP_BUG）は最後まで0件**でした。アプリ本体・テスト用データ（seed）・DBの設計（schema）は**一切変えずに**全緑に到達しています。
- もう一つの大事な結論: 失敗していたのは在庫・イベント・経営ダッシュボード・資金繰りなどの**表示画面だけ**で、機密・同意・CRM閲覧統制（Phase 3 の安全の要）のテストは**最初から最後まで全部合格**。最終的に「スタッフに原価・粗利の金額が漏れず、権限メッセージだけが出る」ことも CI 上で実証されました（機密漏えいなし）。

## 得られた教訓（docs に実際に書かれているもの）

- **tests-only で直す**: 15本すべて、アプリ・seed・schema・CI設定を変えずテストファイルの修正だけで緑化できた。「テストが赤＝アプリが悪い」とは限らない。
- **success という結論だけでなくログ本文で確認する**: 「72 passed (1.1m)」というログの実文や「失敗時だけ動くアップロード工程が skipped だった」ことまで読んで、全緑の証拠を固めた（Evidence Map 方式）。
- **artifact に頼らずログで診断する**: 証拠ファイルが取得不能（proxy 403）でも、CI ログの本文（strict-mode 違反の実要素引用など）だけで4件を確定できた（doc138 の log-based 分析）。
- **分類は「暫定」と明記し、証拠が届いたら潔く訂正する**: 「残りは全部データ不足」という見立ては2回訂正され、最終的に C=0・D=0 だった。証拠（画面の中身）を見るまで断定しない。
- **一歩ずつ直し、毎回 CI 実測で「改善＋悪化ゼロ」を確認する**: F1→F1b→F1c→F1d→F1e と小刻みに進め、57/15→59/13→60/12→62/10→66/6→72/0 と退行なしで積み上げた。
- **診断基盤（F2）を先に整える**: 失敗時にスクショ・操作記録が残る設定を最小限で追加したことが、後の最終確定（doc141）を可能にした。
- **安全の要は常に別枠で見る**: どの回でも機密・同意・CRM閲覧統制のテストは緑で、機密漏えいの証跡ゼロを毎回確認した。

## 変わらない約束

- **AIは外部送信・承認・削除を持たない。** 修復の全期間、封印は維持（外部送信オフ・FakeLLM・外部AI公開は既定オフ）。実LLMもAIコストも使っていない。
- 修復は全編で **schema変更なし・migrationなし・RBAC変更なし・runtime解禁なし**。危険な近道（ネットワーク制約の回避・force push 等）はしていない。
- **テスト全緑＝Phase 3 GO ではない。** 技術ゲート充足後も、進むかどうかは人間の最終 Phase Gate 承認が必須（この判断は [[Phase3最終Gate判断シートとGO記録]] へ続く）。
- GitHub（docs/audit）が正本、Obsidian はナレッジ。
