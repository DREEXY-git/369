# 13. 知財 / moat 戦略（Candidate）

> 種別: **docs-only / strategy candidate**。これは**法的判断ではない**。正式出願は弁理士・弁護士確認が必須。
> 状態: **Candidate**。出願前に公開範囲を確認し、特許化する情報と営業秘密にする情報を分ける。

## 1. 目的

知財を「守り」として整理し、**知財 × データ × 実行基盤 × 開発者経済圏 × 価格戦略 × テンプレート資産**の組合せで moat（堀）を作る方針を候補として固定する。

## 2. 守るべき中核（moat の定義）

テンプレート・ブロック・コードで 369 専用 AI社員を作成し、その AI社員が **369 Runtime / Company Brain / Tool API / Permission Protocol / Evaluation Framework / Billing Meter** に依存して稼働し、制作時と稼働時の両方で従量課金され、開発者と 369 に収益分配される「AI社員 開発・流通・実行基盤」。

## 3. 防御層

特許 / ビジネス関連発明 / 商標 / 営業秘密 / Company Brain / Runtime依存 / Marketplace / テンプレート資産 / 認定制度 / 価格戦略。

## 4. 商標候補（Candidate・要専門家確認）

369 AI社員 / 369 AI補助社員 / 369 AI Employee Studio / 369 Agent Runtime / 369 Company Brain / 369 AI社員マーケット / IKEZAKI OS / IKEZAKI Workforce OS。

## 5. 特許候補（Candidate・要弁理士確認）

- Agent Manifest 標準仕様 / Company Brain スコープ制御 / 3値権限プロトコル / 稼働単位ロイヤリティ分配 / Tool Manifest 標準化 / 評価&改善自動ループ / Safety Review バッジ制度 / 監査ログ改ざん耐性設計 / テンプレ×ブロック×コードの統合UX / 369専用AI社員 作成〜稼働 一貫プロセス。

## 6. 営業秘密として守る候補

- Company Brain の内部スコープ制御ロジック・評価改善の内部指標・課金メータの内部式・審査基準の詳細。
- **注意**: 特許化（公開）と営業秘密（非公開）は排他。どちらにするかを出願前に人間・専門家が判断。

## 7. やらないこと

- 実出願・実登録・法的確約・公開範囲の確定。**法的判断は行わない**。

## 8. 承認ゲート / リスク

- 承認: 出願・公開・商標登録は弁理士/弁護士確認＋人間承認。
- リスク: 先行公開による新規性喪失／営業秘密の意図せぬ開示／商標の抵触。→ 公開範囲確認・専門家確認で緩和。

## 9. 次アクション / 人間判断

- 特許化する項目と営業秘密にする項目の仕分け（人間＋弁理士）。
