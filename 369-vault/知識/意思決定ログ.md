## UsageEvent は「非課金台帳」に徹する

- **判断**: 利用量は貯めるが、課金・決済・請求額計算・cap は実装しない。`billing=usage_only`、金額カラムを持たない。
- **理由**: 経営・お金に触れる領域で、収益化を急ぐと壊す。まず「安全に数える」土台を固め、実課金は Phase 1 完了後・別設計・人間承認で。
- **結論**: emit は8種類まで拡張し、各々本番確認 GO。可視化は read-only の `/admin/usage` のみ。→ [[UsageEvent非課金台帳]]

## worker 経由の emit は「別 helper」で解く

- **判断**: apps/web 専用の `recordUsageEvent`（`@/lib/db` 依存）を worker/packages から使えないため、`packages/db/src/usage.ts::recordUsageEventCore`（`./client` 依存・worker-safe）を新設。
- **理由**: `@/*` エイリアスは apps/web ルート限定。worker から apps/web helper を import できない。
- **結論**: Webhook success（outbox.ts）と worker 朝礼AI出力（jobs.ts）を Core 経由で emit。両 helper とも例外を投げず二重計上を unique 制約で防止。

## EXPORT_JOB / JobRun emit は HOLD

- **判断**: worker `EXPORT_JOB` は enqueue 経路が無く未到達 → 計測しても意味がないので HOLD。JobRun emit は内部インフラ・tenantIdなし・Webhookと二重計上になるため HOLD。
- **理由**: 「走らない処理」「二重計上」を台帳に混ぜると監査証跡が濁る。実利用のエクスポートは既存2経路で計測済み＝漏れではない。
- **結論**: 詳細は `369/docs/audit/20_export_job_trigger_audit.md` / `18_jobrun_usage_event_emit_design.md`。

## 本番確認は「利用者の実測値だけ」で記録

- **判断**: AIは本番に到達できないので、GO/HOLD は利用者が Vercel/CI/本番画面で見た実測値だけを根拠にする。空欄・未確認は HOLD。
- **理由**: 捏造を防ぐ。証拠レベル Level 4 を満たさない限り GO を書かない。
- **結論**: → [[本番確認記録プロンプトの型]]

## docs の状態表現は「git を正」にして陳腐化を防ぐ

- **判断**: `CURRENT_STATE.md` に現在の HEAD/origin/main/branch/未push を固定値で書かない。固定してよいのは基準 commit（実装／完了基準）だけ。
- **理由**: そのファイルを更新した瞬間に固定値が古くなる自己矛盾を避ける。
- **結論**: 現在値は `git rev-parse` 等を正とする。役割分担＝PROGRESS(履歴)/CURRENT_STATE(現在地)/emit matrix(一覧)。→ [[状態管理修復プロンプトの型]]

## 369-vault（本ヴォルト）の新設

- **判断**: コード本体 `369` とは別に、思想・プロンプト・知識を Obsidian 前提の Markdown で貯める `369-vault` を作る。
- **理由**: 「なぜ（why）」は消えやすく最も価値がある。コードの外に頭脳を置く。
- **補足**: セッションの GitHub 連携トークンが `369` スコープのみで**新規 private リポジトリを作成できなかった（403）**ため、初期構築は `369` リポジトリ内 `369-vault/` フォルダとして行い、独立リポジトリへは後日移行する方針。移行手順は [[README]] 併記予定。
