## なぜ次の機能の前に品質基盤なのか

- これまでの安全性は「作るときの規律」（毎回のゲート検査・承認フロー・検証コマンドの手動実行）で守られてきた。全フェーズ green だったが、**この規律を知らない人（または別のAI）が触った瞬間には何のチェックも走らない**。
- Phase 2-B が完了し、守るべき資産（会社の頭脳3テーブル）が本番で確定した今が、**「自動で守られ続ける仕組み」に切り替える最適なタイミング**。次の領域（Case Study / Customer Pain）は個人情報・許諾・外部公開に近づくため、その前に品質ゲートを敷く。

## 入れるものは2つ

1. **CI（自動チェック）**: GitHub にコードが上がるたびに、テスト211本・型チェック・lint が自動で走る。壊れた変更は main に入る前に赤信号で止まる。3段階で導入（まず軽いチェック→ビルド→ブラウザテストは後日判断）。
2. **否定系テスト**: 「できること」ではなく「**できてはいけないことが、本当にできないこと**」を自動で確かめるテスト。8つの対象を優先順で設計した（AIが書き換えられない・権限のない人は拒否される・機密ラベルは2択だけ・外部AI送信は封印・物理削除は存在しない、など）。

## 調査で見つかった一番大事なこと

**「AIは営業プレイブック等を書き換えられない」という約束を守っているのは、たった1つの層（actions 層の isHumanUser チェック）で、そこに自動テストが1本もない。**
権限表の上では AI社員も「下書き作成」の権限を持っている（他機能用）ため、この層が唯一の砦。人間のレビューだけに頼っている状態なので、否定系テストの最優先対象にした。
