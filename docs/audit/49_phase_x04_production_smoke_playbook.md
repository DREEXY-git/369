> docs-only / 本番確認の「型」を固定するプレイブック。**コード・DB・schema・migration の変更は一切なし。本番環境にも触れていない。**
> フェーズ: Phase X-04（Phase X の任意品質資産・doc26 §で予告されていた「本番スモーク定型化」）/ 現在位置は git refs を正とする。

## 1. 非エンジニア向け要約

- これは、**新機能を本番で確認するときの「決まった手順書」**です。今後の Phase 2-B・3c-5・ENSHiN OS・Phase 8 前の確認は、この型に従います。
- あなた（利用者）は **5〜10分の実測**をして、結果をチャットで送るだけ。Claude Code は実測値**だけ**に基づいて GO / HOLD / STOP を記録します。
- Phase 2-A で2回起きた HOLD（どちらもコードのバグではなかった）から得た教訓を、再発しない仕組みとして固定します。

## 2. 背景: Phase 2-A のHOLDから得た教訓

| HOLD | 症状 | 実際の原因 | 学び |
|---|---|---|---|
| 2-A-3a（doc37→doc38） | ナビと新画面が本番で見えない | キャッシュ/反映タイミングの可能性が高い（ハードリロード後の再実測で GO） | 初回NGはまず**ハードリロード・開き直し**で再実測 |
| 2-A-3c-2（doc46→doc47） | AI回答と参照セクションが出ないように見えた | **本番に seed 由来のテストデータが存在しなかった**（データ前提差）。AI回答は実は表示されていた | **本番に実在するデータで確認する**。画面の実文言を記録する |

共通の学び: **HOLD は失敗ではなく、誤修正を防ぐ安全装置**。2回とも慌てて修正していたら、壊す必要のないコードを触っていた。

## 3. 本番確認の絶対原則

1. **本番確認は利用者の実測のみ**。**AI が本番接続確認したものではない**という注記を、すべての本番確認記録に必ず入れる。AI は本番DB・本番画面・Vercel 管理画面を直接見ない・見たことにしない。
2. **本番に実在するデータで確認する**。ローカルの seed（デモデータ）前提で確認項目を設計しない。
3. **§0 利用者実測値テンプレートが未記入なら、Claude Code は記録を作らず停止する**（捏造禁止）。
4. **NG が1つでもあれば GO ではなく HOLD 記録**。NG があること自体を理由に停止せず、HOLD として正確に記録する。
5. **原因未特定なら修正せず read-only 調査**。docs-only 記録とコード修正は必ず別ミッション・別承認に分ける。
6. **HOLD 記録は消さない**。解消は追記（新しい doc＋doc14 の新しい節）で行う（追記主義）。
7. **main push は常に別承認**。

## 4. §0 利用者実測値テンプレート（標準形）

今後の本番確認プロンプトの §0 は、次の型を使う。**「ここに◯◯を記入」型の角括弧プレースホルダ・選択肢の列挙のままの回答欄・日付の「DD」残りなど、テンプレート文字が1つでも残っていたら、Claude Code は編集せず停止**し、チャットでの実測値提出（自由文で可）を案内する。

```
PROD_DATE: （実測日。例: 2026-07-10）
PROD_VERCEL_STATUS: （Ready / green など）
PROD_LATEST_DEPLOY_COMMIT: （Vercel 表示の commit）
PROD_TARGET_DATA_EXISTS: （確認対象データが本番に実在するか。無ければ NOT_PRESENT）
PROD_NEW_FEATURE_CHECKS: （新機能ごとの OK / NG。NG の場合は「画面に実際に何が出ていたか」の文言・状態を必ず添える: エラー文言 / 空白 / 読み込み中 / 別の内容が出た 等）
PROD_EXISTING_SCREENS: （既存画面の無回帰 OK / NG）
PROD_NO_EXTERNAL_SEND: （外部送信・SNS投稿・口コミ投稿・顧客の声公開が発生していないこと）
PROD_AI_NON_CONFIRMATION_NOTE: はい（固定。「AIが本番接続確認したものではない」ことの確認）
PROD_USER_NOTES: （総括・自由文）
```

- 実測値はチャットの自由文でもよい（前例: 2-A-3b-1 以降はチャット提出で成立）。Claude Code が §0 に整形して記録する。
- NG の報告では「表示されない」だけでなく、**実際に何が表示されていたか**を書く（2-A-3c-2 の教訓: 「AI回答が出ない」の実態は「既存ナレッジ由来の回答は出ていた」だった）。

## 5. GO / HOLD / STOP 判定ルール

| 判定 | 条件 | Claude Code の行動 |
|---|---|---|
| **GO** | 事前に列挙した確認項目が**すべて OK**＋AI非確認注記=はい | GO 記録（新doc＋doc14 追記＋基準更新）。基準 commit は**アプリ実装 commit**（docs commit ではない） |
| **HOLD** | NG が1つでもある（既存画面は正常でも） | HOLD 記録を正確に作る（GO済み基準は**動かさない**）。次は read-only 原因調査。**GO にできないが既存画面は正常、は典型的な HOLD**であり停止理由ではない |
| **STOP（停止）** | §0 がテンプレート未記入／git 期待値不一致／仕様曖昧／禁止領域（DB・認証・課金・外部送信等）に踏み込む必要が出た | **何も編集せず停止**して実測値・判断材料を人間に要求する。推測で埋めない |

- 判定条件（何が OK なら GO か）は、**本番確認プロンプトに事前に明記**しておく（2-A-3c-2-PROD-2 の判定ルール方式）。
- HOLD 解消は「read-only 調査 → 仮説 → 利用者再実測 → 解消GO記録」の順（doc38・doc47 の前例）。

## 6. 本番に実在するデータで確認するルール

- 確認項目を設計する段階で、「その画面・その回答に必要なデータは本番に存在するか？」を必ず問う。
- seed はローカル専用。**本番では seed は自動実行されない**（既知の前提。doc46 §3）。
- 対象データが本番に無い場合の期待挙動（空一覧・参照セクション非表示など）を、**事前に「正常」と定義**しておく（2-A-3a の「一覧が空でも正常」の前例）。

## 7. テストデータ作成・片付けルール

- テストデータが必要な場合は、**本番UIの既存GO済み機能**（人間用の作成画面）で利用者が作成する。AI・スクリプトによる本番DB直接操作は行わない。
- 確認が終わったテストデータは、**アーカイブ機能で片付ける**（物理削除はそもそも存在しない設計）。実データとして残す価値があるもの（例: 2-A-3c-2 の「値引き承認ルール」会社方針）は、利用者判断でそのまま運用に乗せてよい。
- テストデータに **PII・実顧客名・実価格・secret を入れない**。

## 8. docs-only 記録とコード修正を分けるルール

- 本番確認の記録（GO/HOLD）は **docs-only ミッション**。同じミッションでコードを直さない。
- 原因調査は **read-only ミッション**。調査中にコード修正が必要と判明しても、その場で直さず「最小修正案」の提示まで。
- 修正は**別承認の実装ミッション**で行い、修正後に再実測 → 記録、の順に戻る。
- この分離が、2-A の2回の HOLD で「修正不要のコードを触らない」ことを保証した。

## 9. AIが本番確認したことにしないルール

- すべての本番確認記録に定型注記を入れる: **「本確認は利用者の Vercel 画面・本番画面の実測によるものであり、AI が本番接続確認したものではない。」**
- AI は本番の URL を開かない・本番DBに接続しない・Vercel ダッシュボードを見ない。
- 利用者の包括報告（「その他はOK」）はそのまま記録してよいが、**包括報告である旨を注記**し、論点に関わる項目は個別再確認の対象にする（doc46 の ai_reference ログ注記の前例）。

## 10. ENSHiN OS / 外部発信 / 口コミ / SNS / 顧客の声の追加停止条件

ENSHiN OS（関係資産OS）領域の本番確認・実装では、上記に加えて以下を**無条件の停止条件**とする。

- **口コミ投稿をしない**（Google プロフィール等への投稿・返信の自動実行を含む）。
- **SNS投稿をしない**（下書き作成まで。公開ボタンは人間）。
- **顧客の声を許諾なしに公開しない**。**推薦コメントを許諾なしに掲載しない**（`ConsentRecord` 等による**許諾**の記録が確認できるまで、公開系は一切進めない）。
- **社長ブランディング投稿・会社ブランディング投稿を自動公開しない**。
- **外部発信は「下書き作成・管理・承認フロー・リスク検知・許諾管理・ログ記録」まで**。実際の発信は人間承認＋実測確認後のみ。
- **広告表現・No.1表記・効果保証・誇大表現**の疑いがある文面は、AI が判定せず**人間・法務判断へ送る**（リスク指摘までが AI の役割）。
- 確認中に外部発信が「発生していないこと」自体を、§0 の確認項目（PROD_NO_EXTERNAL_SEND）として毎回実測する。

## 11. Phase 2-B 以降で使う標準プロンプト骨子

今後の本番確認プロンプトは、次の骨子を使う（2-A で実証済みの型の一般化）:

1. **ヘッダ**: フェーズ名・対象 commit・docs-only 宣言。
2. **§0 利用者実測値**（本書 §4 の型。未記入なら停止）。
3. **Expected values**: EXPECTED_LOCAL_HEAD / REMOTE_MAIN / FEATURE_HEAD / HEAD_SUBJECT / 未push / NEXT_AUDIT_DOC_NUMBER / doc14 節番号。
4. **ALLOWED_FILES**（通常6ファイル: 新doc・doc14・CURRENT_STATE・PROGRESS・vault note・vault index）／**FORBIDDEN**（apps/・packages/・schema・migrations・seed・rbac・labels・package/lock・.env・vercel.json・既存doc本文）。
5. **判定ルール**: GO にしてよい条件の全列挙＋NG 1つで HOLD＋テンプレ未記入で STOP（本書 §5）。
6. **手順**: Scout（git 期待値・重複確認）→ §0 検査 → 記録作成 → post-edit gates（allowed exact / forbidden zero / doc14 append-only・節一意 / 必須語句 / stale・template / diff --check / secret scan 値非表示）→ 全green時のみ commit → push 規定（feature のみ or 禁止。main は常に別承認）。
7. **16項目最終報告＋最新値ブロック＋必須文言**。
8. **ENSHiN OS 領域なら本書 §10 の停止条件を追加**。

## 12. 今回やっていないこと

- **本番自動監視**（アラート・死活監視）: 未実装・後続別承認。
- コード変更・DB変更・本番接触・外部送信・**Phase 8**（課金）・MCP/API公開・ENSHiN OS 外部発信: **一切なし**。
