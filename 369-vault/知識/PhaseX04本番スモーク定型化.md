## 結論（1行）

**本番確認のやり方を「手順書（プレイブック）」として固定した — 5〜10分の実測で、捏造ゼロのまま GO / HOLD / STOP を正確に記録できる型（Phase X-04・docs-only・判定 GO）。**

## なぜ作ったか（非エンジニア向け）

- Phase 2-A で HOLD が2回起きた。どちらも**コードのバグではなかった**（キャッシュ疑い・本番にデータが無かっただけ）。
- つまり「機能は正しいのに NG に見える」ことが実際に起きる。次のフェーズ（Phase 2-B・ENSHiN OS・課金前）ではもっと重要になるため、確認の型を先に固定した。

## プレイブックの柱（doc49 の要約）

1. **本番確認は利用者の実測のみ** — AI は本番を見ない・見たことにしない（全記録に注記必須）。
2. **§0 実測値テンプレート** — 未記入なら Claude Code は何も書かずに停止（捏造禁止）。NG は「何が表示されていたか」の実文言まで記録。
3. **GO / HOLD / STOP の3分岐** — 全部OKなら GO。NG 1つでも HOLD（既存画面が正常でも）。テンプレ未記入・期待値不一致なら STOP。
4. **本番に実在するデータで確認** — デモデータ前提で確認項目を作らない。必要なら本番UIのGO済み機能でテストデータを作り、終わったらアーカイブで片付ける。
5. **記録と修正を分ける** — 記録は docs-only、調査は read-only、修正は別承認。この分離が誤修正を防いだ実績がある。
6. **ENSHiN OS 追加停止条件** — 口コミ投稿・SNS投稿・顧客の声公開・推薦コメント掲載・ブランディング投稿は自動でやらない。公開系は**許諾**の記録が確認できるまで進めない。広告表現・No.1・効果保証は人間・法務判断へ。
