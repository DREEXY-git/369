## なぜこの役割分担があるのか

「push未実施」「人間承認待ち」のような**その瞬間しか正しくない状態**を永続文書に書くと、次の作業で古くなり現在地が分からなくなる。これが過去の品質低下の主因だった。そこで「どの文書に・何を・いつ書くか」を1回だけ決めて固定した。

## 役割の一覧（迷ったらここ）

| 文書 | 役割 | ひとこと |
|---|---|---|
| `tasks/PROGRESS.md` | **履歴** | 何をやってきたか（Phaseごとの確定事実） |
| `tasks/CURRENT_STATE.md` | **現在地** | 今どこにいるか（1枚サマリー） |
| `docs/audit/usage_event_emit_matrix.md` | **emit一覧** | UsageEvent 8種類の正本 |
| `docs/audit/14_release_stabilization.md` | **本番確認** | 利用者実測に基づく GO/HOLD |
| `docs/audit/15_monetization_usage_design.md` | **詳細設計** | UsageEvent/Monetization の設計史 |
| `369-vault/` | **思想・プロンプト・知識** | なぜそうするか（why） |

## 大原則

- **現在の git 反映状態は git refs が正**（`git rev-parse` / `git log origin/main..HEAD` / `git status`）。文書に現在HEADの固定値を書かない。固定してよいのは「実装 commit」「完了基準 commit」だけ。
- **一時状態（push未実施・承認待ち等）を永続docsに書かない。** 必要なら最終報告で扱う。
- **secret / PII / 本文 / raw metadata / 金額 / 本番データを書かない。** これは全文書共通（vault 含む）。
- **未確認のGOを書かない。** 本番確認は利用者の実測値だけを根拠にする（→ [[本番確認記録プロンプトの型]]）。
