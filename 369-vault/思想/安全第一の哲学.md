# 安全第一の哲学

> 目次に戻る → [[index]] ／ 関連 → [[369の思想と世界観]] ・ [[AIの役割と境界]] ・ [[セキュリティと権限]]

## 原則：速さより「壊さないこと」

このプロダクトは経営・お金・個人情報・外部送信・本番環境に触れる。だから開発の全工程で **「壊さない」ことを速度より優先**する。この哲学は、実際の開発フロー（フェーズ運用）に具体化されている。

## 証拠主義（Evidence over "完了しました"）

- **「完了しました」だけを信じない。** 検証していないものを成功扱いしない。
- 変更したら必ず、そのプロジェクトに実在する検証コマンド（lint / typecheck / build / test / Prisma generate）を実行し、**通ったものだけを「通った」と言う**。
- 本番確認は、**利用者が実際に Vercel / CI / 本番画面で見た実測値だけ**を根拠にする。AI はサンドボックスから本番に到達できない。
- **値を捏造しない。** 未確認を GO にしない。実測値が空欄なら **HOLD で止まる**。

## 証拠レベル（判断の物差し）

| レベル | 意味 |
|--------|------|
| Level 0 | 未確認（AIが言っているだけ） |
| Level 1 | 作業痕跡あり（変更はあるが検証なし） |
| Level 2 | 静的検証済み（lint / typecheck / build のどれか） |
| Level 3 | 動作検証済み（test / E2E / 画面確認 / API確認） |
| Level 4 | 安全確認済み（権限・テナント分離・個人情報・外部送信・本番影響まで） |

権限・個人情報・本番画面に関わる変更の GO 記録には **Level 4 相当**が要る。

## フェーズ運用（安全を工程に落とす）

1つの大きな作業を、**目的が1つだけの小さなフェーズ**に割る。各フェーズは別プロンプトで、明示的な「やること／やらないこと／停止条件」を持つ。典型的な並び：

1. **設計・監査（docs-only）** — 読み取り専用。コードを触らず、方針だけ決める。
2. **最小実装＋検証＋ローカルコミット** — 薄い縦切りで実装し、検証を通し、**push はしない**。
3. **push（別承認）** — fast-forward のみ。force push 禁止。→ [[push専用プロンプトの型]]
4. **本番確認記録（別承認）** — 利用者の実測値だけで GO/HOLD を記録。→ [[本番確認記録プロンプトの型]]

## 危険領域では「実装せず停止」

以下に影響するなら、AIの自己判断で進めず、**実装せず停止して報告**する：
DB・認証・課金・個人情報・外部送信・本番環境・契約・請求・会計・給与・録音・位置情報。

## 修正ループの上限

- エラーが出たら最大3回まで修正ループを回す。
- 同じエラーが2回以上続く／3回失敗したら、**止めて**「試したこと・残る原因候補・人間に確認すべき点」を非エンジニアにも分かる言葉で報告する。

## push の作法

- 常に `git push -u origin <branch>`（作業ブランチ）か、記録系は `git push origin HEAD:main` を明示。
- **force push 禁止・rebase / reset / amend 禁止。** 戻すときは `git revert`（前進的取り消し）。
- push は原則そのつど人間承認。承認は1回きりで、次のフェーズには持ち越さない。

## 非エンジニアへの説明責任

- ユーザーはコードを読めない。だから報告は**非エンジニアにも判断できる言葉**で、証拠付きで書く。
- 「何をしたか・検証は通ったか・pushしたか・次に何をすべきか・人間が判断すべきことは何か」を必ず添える。
