# 本番確認記録プロンプトの型

> 目次に戻る → [[index]] ／ 関連 → [[安全第一の哲学]] ・ [[フェーズ実装プロンプトの型]]

## 用途

実装を push した後、**利用者が実際に見た本番/CIの実測値だけ**を根拠に、GO または HOLD を docs に記録させる型。**捏造を構造的に防ぐ**のが狙い。

## 最重要ゲート

- 実測値の貼り付け欄に **空欄・未確認・NG・矛盾が1つでもあれば、ファイルを一切変更せず HOLD で停止**。
- 「全部GOです」だけでは不十分。**各項目に具体値**（`b08c939` / `Ready` / `成功` / `表示されない` / `GO` 等）が要る。
- AI は本番に到達できない。だから **AI が値を推測して GO を書いてはいけない**。

## 実測値欄（例：read-only 画面の確認）

```
Production Branch:
Latest Production Deployment Commit:
Implementation Commit:
Status:                 （Ready か）
Build:                  （成功か）
Runtime error:          （なし か）
/画面 が表示できる:
権限ありユーザーで表示される:
権限なしユーザーでは隠れる:
表示は許可した項目だけ:
個人情報 / 金額 / secret 実値が表示されていない:
（安全境界：emit対象数・課金なし・決済なし・本番DB操作なし …）
総合判定: GO / HOLD / NG
```

## OK とみなす値 / HOLD にする値

- **OK**: OK / GO / なし / 成功 / Ready / main / 該当コミットSHA / 不要 / 表示される / 表示されない / 守られている。
- **HOLD**: 空欄 / 未確認 / 不明 / NG / たぶん / おそらく / 見ていない / 矛盾する値。

## GO のときだけ記録

- 監査ドキュメント（例 `docs/audit/14_release_stabilization.md`）に新セクションで記録。
- 「本確認は利用者の Vercel/CI/本番画面確認によるものであり、AIが本番接続確認したものではない」を必ず添える。
- 一時状態（`push未実施` / `人間承認待ち`）を**永続docsに書かない**。反映状態は git refs を正とする。→ [[状態管理修復プロンプトの型]]

## コツ

- 「secret という**単語**が注意文にある」のは OK、「secret の**実値**が表示」は NG。単語と実値を区別させる。
- コード側の安全性は grep で**再監査**させ、実測値（本番）とコード（静的）の**両方**が揃ってはじめて GO。
