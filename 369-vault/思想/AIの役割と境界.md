# AIの役割と境界

> 目次に戻る → [[index]] ／ 関連 → [[安全第一の哲学]] ・ [[セキュリティと権限]] ・ [[369の思想と世界観]]

## AIに「させること」

- 現状監査・設計判断・実装・検証・リスク検知・ドキュメント更新・非エンジニア向け報告を**一貫して**担う。
- AIの生成物（営業文・レポート・分析）は**必ず下書き（draft）**として作る。
- AI回答には可能な限り **参照元／根拠／信頼度／推奨アクション**を残す。

## AIに「絶対にさせないこと」

- **外部送信・承認・削除の権限を AI に持たせない。** `ROLE_PERMISSIONS` の `AI_AGENT` / `AI_ASSISTANT` を変更しない。
- 外部送信は `EXTERNAL_SEND_ENABLED=true` かつ **人間承認時のみ**。
- **断定助言をしない**: 法務／税務／労務／財務を断定せず、リスク・確認観点・専門家相談候補に留める。
- **本番確認値を捏造しない**: 見ていない本番結果を GO と書かない。

## なぜ境界を引くのか

- AIは強力だが、経営・お金・個人情報・外部送信を**取り返しのつかない形で**動かしうる。
- だから「AIは下書きまで、実行の引き金は人間」という構造を、**権限（RBAC）とワークフロー（承認）で物理的に**固定する。単なるお願いにしない。→ [[セキュリティと権限]]

## 個人情報・機密の扱い

- 外部LLMへ送る前に `maskText` でマスク可能。機密参照（AI含む）は `writeDataAccess` で記録。
- 利用量記録（UsageEvent）の metadata には **本文・顧客情報・メール・金額・secret・保存先パス・実IDを入れない**。入れてよいのは固定の非PIIラベルだけ。→ [[UsageEvent非課金台帳]]

## AI Provider の設計

- `getLLMProvider() / getEmbeddingProvider() / getTranscriptionProvider()` で取得。env 未設定なら自動で **Fake** にフォールバック。
- 各AIタスクには `fakeX()` 実装があり、実LLM失敗時も fake に落ちて**壊れない**。
- FakeLLM でも入力データに基づく「それらしい日本語」を返す。出力は Zod スキーマで検証。

## 将来のAI社員像（ビジョン）

- 補助型AI → 自立型AI → 管理職AI → 経営AI と段階的に責務が上がる構想。
- ただし**責務が上がっても、外部送信・承認・削除・断定助言・本番確認の捏造という境界は動かさない**のが原則。
