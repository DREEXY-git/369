# push前敵対的レビューの型

> 目次に戻る → [[index]] ／ 関連 → [[実装前Gateプロンプトの型]] ・ [[P3CT4FakeLLM下書き生成]] ・ [[安全第一の哲学]]

## 用途

commit 済み・**push 前**のコードに対して、複数視点の独立 AI レビューで「本当に穴がないか」を総点検させる型。P3-CT-4 で実際に **high 1件（メモ表示経由の finance 件数漏れ＝redaction 迂回）を push 前に発見・修正**し、外部影響ゼロで閉じた実績がある。

## 骨子

```
# 0. 目的
未push commit <sha> を、独立した複数視点で敵対的にレビューする。
各視点は互いの結果を見ない（独立検出）。見つけた指摘は別の検証者が
「本当に再現するか」を反証を試みる形で確認してから確定する。

# 視点の例（6視点）
1. 権限・RBAC 迂回（AIロール・権限なしロールで実行できないか）
2. redaction 迂回（隠したはずの値が別経路で見えないか。表示側・生成側の両方）
3. 入力検証・インジェクション（allowlist 外の値・プロンプト注入）
4. 監査ログ・記録漏れ（重要操作に帳簿が付いているか）
5. PII・secret 混入（ログ・metadata・AI入力に入っていないか）
6. e2e 回帰（既存テストの文言カウント・セレクタを壊さないか）

# 確定と修正
- 複数視点が独立に同じ指摘をしたものは最優先（P3-CT-4 の high は6視点全てが検出）
- 確定した high は push 前に修正し、その穴を CI で恒久的に見張る自動テストを追加
- 修正も含めて再検証（単体・型・lint・安全ゲート）してから push 承認へ

# 教訓（P3-CT-4 の実例）
生成側で redaction を守っても、**表示側のクエリが閲覧者の権限を見ていなければ迂回される**。
防御は「書くとき」と「見せるとき」の両方に置く（二重防御）。
```

## 効果

- push 前に見つければ外部影響はゼロ。レビューコストは漏えい対応コストより常に安い。
- 「穴を塞ぐ」と「穴を見張るテストを足す」をセットにすることで、同じ穴が二度と開かない。
