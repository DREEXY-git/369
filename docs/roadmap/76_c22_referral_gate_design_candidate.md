## 1. スコープ（v0 で作るもの / 作らないもの）

### 作るもの（将来の実装 WIP・本書では設計のみ）
- **read-only の紹介台帳可視化**: 紹介関係・紹介コード・成果候補・承認状態の閲覧（marketing:read）。
- **AI の関与は下書きまで**: 紹介お礼文・案内文の下書き（FakeLLM 直呼び・Zod・外部送信なし）。

### 作らないもの（恒久的な設計制約）
- 自動報酬支払い・ポイント自動付与（報酬は**候補**の算出まで。支払いは人間承認＋既存の承認導線）。
- 紹介リンクの外部配布自動化・SNS 自動投稿（外部送信封印の対象）。
- 顧客 PII の紹介レコードへの複製（次項）。

## 2. データ設計方針（schema は未実装・設計のみ）

| 要素 | 方針 |
|---|---|
| tenantId | 全テーブルにスカラ必須・全クエリ tenant スコープ（既存規約） |
| 紹介者（referrer） | **Customer への参照 ID のみ**を持ち、氏名・連絡先を複製しない（PII 非複製。表示時に権限ゲート越しで JOIN） |
| 紹介コード | ランダム生成・推測不能・tenant 内 unique・失効日時（expiresAt）必須 |
| 成果候補（conversion candidate） | 「紹介経由の商談/成約らしい」イベントを**候補**として記録（確定は人間）。根拠（どのイベント由来か）を必須保持 |
| 承認 | 報酬候補の確定は `ApprovalRequest`（既存）経由。**AI・自動処理は承認しない** |
| 不正検知 | 自己紹介（自分を紹介）・同一連絡先の循環・短期大量発行を**候補フラグ**として表示（自動 BAN しない・人間判断） |
| 保持期間・削除連動 | 顧客削除時に紹介レコードの個人参照を切断（ID 参照のため削除カスケード or NULL 化で対応）。保持期間は同意記録（ConsentRecord）と整合させる |
| migration/rollback | 追加時は additive-only（既存テーブル変更なし）・rollback は drop のみで完結する独立テーブル群とする |

## 3. コンプライアンス制約（実装 DoR に含める）

- **ステマ防止**: 紹介による対価が発生する場合、紹介を受ける側への**関係性明示（PR 表記）**を文面テンプレートに必須で含める（AI 下書きテンプレートにも固定文言として埋め込み・削除不可の設計）。
- **共有同意**: 紹介者が被紹介者の情報を持ち込む場合、`ConsentRecord` に同意記録がない連絡先へは接触しない（`isSuppressed` 併用・既存のメール営業ルールと同一）。
- **報酬の税務**: 報酬候補の表示に「税務上の取扱いは専門家確認が必要」の注記（AI は断定助言しない）。

## 4. DoR / DoD / E2E / 停止条件（実装 WIP 用）

- **DoR**: 本書承認＋schema 追加の明示承認＋対象テナントのデモデータ設計。
- **DoD**: read-only 台帳＋AI 下書き（封印）＋承認導線接続＋監査ログ＋unit/e2e＋CI green ログ本文確認。
- **E2E 最低線**: 権限なしロールの非表示・紹介コード失効の表示・成果候補が「候補」表示のまま人間承認まで確定しないこと・PR 表記文言がテンプレートから消せないこと。
- **停止条件**: 外部送信の解禁が必要になった時点・報酬の自動支払いが要件化された時点・顧客 PII 複製が避けられない設計になった時点（いずれも人間判断へ差し戻し）。
