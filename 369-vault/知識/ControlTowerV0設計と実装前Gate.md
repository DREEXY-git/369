## これは何か

- Phase 3 の最初の一歩「AI Growth Opportunity Control Tower v0」の**設計図**（doc146）と、**作り始める前の最終チェック（実装前 Gate）**（doc147）の2本セットです。どちらも docs-only で、**コードは1行も書いていません（実装未着手）**。
- Control Tower とは: 会社の既存データ（新規開拓リード・商談・会社の頭脳・Golden Path・資金繰り・利用/監査ログ）から「次に手を打つべき成長機会」を1画面に集め、社長が次の一手を選べる**見るだけ（read-only）中心の管制塔**です。置き場所は `/growth/control-tower`。

## どう作る計画か（やさしい言い換え）

- 出すカードの例: 未追客リード・停滞商談・高機会リード・会社の頭脳の改善提案・低粗利の改善候補・未回収リスク・次回接触推奨・既存顧客への追加提案候補・社長が見るべき成長機会。
- 段組み（実装フェーズ案）: **P3-CT-0 設計 → CT-1 read-only 画面 → CT-2 優先度ロジック → CT-3 監査 → CT-4 AI 下書き → CT-5 承認導線 → CT-6 e2e → CT-7 push/CI/Gate**。各段は別承認・停止（STOP）条件つきで、一段ずつしか進みません。
- 実装前 Gate の一番大事な結論: **v0 は今ある DB のまま・今ある権限のまま・今あるデモデータのままで作れる**（既存 schema・RBAC・seed のみで成立、Gate PASS・STOP 非該当）。見るだけの画面なので、新しいテーブル・権限・デモデータは不要と、実際のコードを調べて確認済み。
- **状態永続化（カードの既読・スヌーズ・ピン留め・並び順固定）は v0 では作りません**。将来必要になったら、その時点で STOP して別承認（DB 設計変更を伴うため）。

## 変わらない約束

- スタッフには原価・粗利などの金額を見せず伏せ字にする既存の仕組みを踏襲。顧客の個人情報（PII）は増やさず、件数など匿名の指標が中心。
- AI はダミー（FakeLLM）で、外部送信・承認・削除・自動実行はしない。提案は「下書き」まで、重要操作は承認申請（ApprovalRequest）経由。
- **外部送信・実 LLM・課金・本番反映は、1つずつ人間が承認するまで行わない**（個別承認制を維持）。
- 実装中に 状態永続化・新規権限・新規 seed・伏せ字で塞げない機密表示・CI 設定変更 が必要になったら、その時点で STOP して別承認。

## 次の一手（すべて人間判断・別承認）

1. P3-CT-1（read-only 画面）の実装 — 既存関数（ログイン確認・ダッシュボード集計・財務伏せ字・監査ログ）を再利用し、既存テストを壊さない → [[P3CT1ReadOnly画面とE2Eセレクタ強化]]
2. その先の CT-2 以降（優先度ロジック・監査・AI 下書き・承認導線・e2e・push）は各段ごとに別承認
3. schema / RBAC / seed に影響が出た場合は実装前に STOP して別の重い承認
