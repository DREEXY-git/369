## 番号が揃っていないように見える理由

「doc105 なのに Phase は 2-A-3c で、基準の commit は 611e51e……番号がバラバラ？」と見えても、**異常ではありません**。これらは**別々の目的で動く別の軸**だからです。無理に揃えると、逆に履歴を書き換えることになって危険です。

## 5つの軸の違い

- **audit番号（doc◯◯）**: 作業した順に増える「時系列の証拠番号」。設計・決定・詳細設計・push だけの回も、それぞれ1番を使う。だから機能が枝分かれしても増え続ける。
- **Phase番号（例: Phase 2-A-3c）**: ロードマップ上の「開発段階」。機能の位置を表すので、audit番号とは一致しない。
- **Lineage（例: Customer Pain）**: ある機能の「一連の流れ」。複数の Lineage が並行するので、1本線の連番にはならない。
- **Stage**: 「作業の種類」（設計／方針決定／詳細設計／実装／本番確認／push だけ、など）。
- **commit hash（例: 1372a35）**: GitHub 正本の「実際の状態」を指す番号。基準（Baseline）の commit とは別。

一般的な開発現場でも、監査ログ・リリースノート・ロードマップ・commit は別々に管理されます。**バラバラに見えるのが自然**です。
