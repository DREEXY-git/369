# 12. PLUG Commerce / 369 Employee App 戦略（Candidate）

> 種別: **docs-only / strategy candidate**。実装・ブラウザ拡張・購買実行・アフィリエイト報酬・個人情報保存なし。
> 状態: **Candidate**。DB化・拡張機能実装・報酬実装は別承認。**個人情報・実顧客/従業員データは扱わない**。

## 1. 目的

導入企業の**法人購買**と**従業員の日常購買**に、透明性と企業統制を武器に安全に入り込む導線を設計候補として固定する（Honey型の不透明リンク差し替え批判を回避）。

## 2. PLUG Commerce（購買エンジン）

- **B2C向け候補**: ブラウザ拡張 / 商品ページ認識(JAN・ASIN・型番) / EC横断価格比較 / 送料・ポイント・クーポン込み実質価格 / 納期・在庫比較 / 価格履歴 / 最安候補 / アフィリエイト計測 / ユーザー還元。
- **B2B/法人購買向け候補**: 会社指定ショップ / 購買ポリシー / 承認購買 / 部署別予算 / 勘定科目候補 / 経費精算連携 / 請求書払い / 法人価格比較 / 購買履歴 / 仕入先評価 / 不正購買検知 / 購買申請→承認→発注→請求候補→会計候補。

## 3. 透明性の設計ルール（必須・Candidate）

- アフィリエイトリンク挿入は**明示**する。
- **ユーザーに利益がある時だけ**表示する。
- 元リンクを**無断で奪わない**。
- どのリンクで誰に手数料が入るかを**透明化**する。
- 企業管理者が利用範囲を制御できる。
- 拡張機能の権限は**最小化**・セキュリティ審査必須。

## 4. 369 Employee App

- 経営者/管理部門/現場社員/従業員個人それぞれに価値（タスク・申請・承認・勤怠候補・FAQ・AI相談・福利厚生・割引・キャッシュバック・最安検索・クーポン・AI生活サポート）。
- 会社モード / 個人モードを分離。

## 5. データ分離（絶対ルール・Candidate）

- **会社購買**: 会社管理者が確認可能。
- **個人購買**: 従業員本人だけが確認。
- **会社への表示は個人を特定しない匿名集計レベルに留める**。
- キャッシュバック: 会社購買分は会社、個人購買分は従業員。
- ブラウザ拡張: 仕事用モード / 個人用モードを分離。

## 6. やってはいけないこと（禁止・恒久）

- こっそりリンクを差し替える / 元リンクを奪う / ユーザーメリットなしに cookie を入れる。
- **私的購買履歴を会社へ見せる** / 拡張機能に過剰権限 / 会社用アカウントで私的購買を強制追跡。

## 7. やらないこと（今回）

- 実装・ブラウザ拡張・購買実行・決済・アフィリエイト報酬・個人情報/購買データ保存・外部送信。

## 8. 承認ゲート / リスク

- 承認: 拡張機能公開・法人購買β・従業員配布・報酬実装は重い個別承認＋法務確認。
- リスク: 個人情報・行動データの取扱い、拡張機能の権限、透明性欠如による信頼毀損。→ データ分離・最小権限・開示・審査で緩和。

## 9. 次アクション / 人間判断

- PLUG を事業Phase 6 として技術検証（PoC・mockのみ）するか（人間判断）。
- 従業員個人データの取扱いは法務・プライバシー確認が前提（人間判断・専門家確認）。
